## HIGH PRIORITY ISSUES (Fix Immediately)

### #1 SQL Injection Risk via LLM-Generated SQL
- **Severity:** CRITICAL
- **Files:** `src/queryease/validator.py`, `src/queryease/executor.py`, `src/queryease/db/*.py`
- **Problem:** The application executes SQL generated by an LLM without parameterization. The validator only checks for forbidden keywords and schema validity—not for SQL injection patterns.
- **Impact:** An attacker who can influence the LLM prompt (prompt injection) could potentially execute arbitrary SQL, including data exfiltration or modification.
- **Solution:** 
  - Implement read-only mode by default
  - Use database user credentials with minimal privileges
  - Add SQL AST analysis for injection patterns
  - Consider prepared statement patterns where feasible

### #2 Credentials Stored in Plain Text .env File
- **Severity:** CRITICAL
- **Files:** `src/queryease/config.py`
- **Problem:** Database credentials and API keys are stored in a `.env` file in plain text with no encryption or secure storage mechanism.
- **Impact:** If the project directory is compromised, all credentials are exposed.
- **Solution:** 
  - Support integration with system keychain (macOS Keychain, Windows Credential Manager)
  - Support environment variable injection from secret managers (AWS Secrets Manager, HashiCorp Vault)
  - Encrypt the .env file or use a secrets manager

### #3 No Input Sanitization Before LLM
- **Severity:** HIGH
- **Files:** `src/queryease/generator.py`, `main.py`
- **Problem:** User questions are passed directly to the LLM without any sanitization or length limits.
- **Impact:** Could enable prompt injection attacks, token exhaustion, or leakage of sensitive information.
- **Solution:** 
  - Implement input length limits (e.g., max 500 characters)
  - Sanitize potentially malicious patterns
  - Add prompt injection detection
  - Log and audit unusual inputs

### #4 Write Operations Lack Transaction Safety
- **Severity:** CRITICAL
- **Files:** `src/queryease/executor.py`, `src/queryease/db/*.py`, `main.py`
- **Problem:** INSERT/UPDATE/DELETE operations execute without transaction wrapping or rollback capability. If an error occurs mid-operation, partial changes may persist.
- **Impact:** Data integrity risk for write operations.
- **Solution:** 
  - Wrap write operations in transactions
  - Provide explicit commit/rollback flow
  - Add dry-run mode to preview changes before execution

### #5 Validator's String-Based Keyword Check is Bypassable
- **Severity:** HIGH
- **Files:** `src/queryease/validator.py`
- **Problem:** The `check_forbidden_keywords` function uses simple string matching (`"DROP" in sql_upper`), which can be bypassed with:
  - Case variations in certain contexts
  - Unicode characters
  - Comment injection (`DR/*comment*/OP`)
  - Encoded payloads
- **Impact:** Dangerous statements could bypass validation.
- **Solution:** 
  - Use proper SQL parsing (sqlparse AST) to analyze statement types
  - Check actual statement types, not keywords
  - Validate the parse tree structure

### #16 Typo in History Save - Wrong Attribute Name
- **Severity:** HIGH
- **Files:** `main.py` (line 211)
- **Problem:** Code uses `result.rowcount` but the `QueryResult` class defines `row_count` (with underscore). This will cause an `AttributeError`.
- **Impact:** History saving will crash for SELECT queries.
- **Solution:** Fix the attribute name to `result.row_count`.

### #17 No Interactive Mode
- **Severity:** HIGH
- **Files:** `main.py`
- **Problem:** Tool only works in one-shot CLI mode. Users must re-type the full command for each query.
- **Impact:** Very frustrating for iterative query development; no REPL mode.
- **Solution:** 
  - Add interactive REPL mode with readline support
  - Support multi-line input
  - Add command history within session

### #37 No Multi-Turn Conversation Support
- **Severity:** HIGH
- **Files:** `src/queryease/generator.py`, `main.py`, `src/queryease/history.py`
- **Problem:** Each query is independent. Can't ask follow-up questions like "now filter to only active users".
- **Impact:** Major UX limitation; users expect conversational interaction.
- **Solution:** 
  - Maintain conversation context
  - Include previous queries in LLM prompt
  - Track session state

### #44 No .env.example File Referenced in README
- **Severity:** HIGH
- **Files:** Create `.env.example`
- **Problem:** README mentions `.env.example` but this file does not exist in the project.
- **Impact:** New users can't set up the project.
- **Solution:** Create `.env.example` with placeholder values for GROQ_API_KEY and DATABASE_URL.

## MEDIUM PRIORITY ISSUES

### #6 No Rate Limiting on LLM Calls
- **Files:** `src/queryease/generator.py`, `src/queryease/judge.py`, `src/queryease/config.py`
- **Problem:** No rate limiting or quota enforcement on Groq API calls.
- **Solution:** Implement request throttling; add daily/monthly call limits; cache LLM responses for identical prompts.

### #7 History Database Stores Raw SQL Without Encryption
- **Files:** `src/queryease/history.py`
- **Problem:** Query history (including potentially sensitive queries) is stored in plain SQLite without encryption.
- **Solution:** Encrypt the history database; add option to disable history; implement history auto-expiry.

### #8 Global State in Generator and Judge Modules
- **Files:** `src/queryease/generator.py`, `src/queryease/judge.py`
- **Problem:** Both modules use module-level `_client` global variable for the Groq client, creating hidden coupling and testing difficulties.
- **Solution:** Use dependency injection; create a client factory class.

### #9 Inconsistent Exception Handling
- **Files:** `src/queryease/generator.py`, `src/queryease/judge.py`, `src/queryease/executor.py`
- **Problem:** Broad `Exception` catches, lost tracebacks, silently swallowed exceptions.
- **Solution:** Use specific exception types; preserve tracebacks with `raise ... from e`; implement structured logging.

### #10 Column Extraction Regex is Fragile
- **Files:** `src/queryease/validator.py`
- **Problem:** Complex regex patterns fail on nested subqueries, complex expressions, window functions, CTEs.
- **Solution:** Use sqlparse AST traversal instead of regex.

### #11 Table Extraction Misses Edge Cases
- **Files:** `src/queryease/validator.py`
- **Problem:** Regex won't handle schema-qualified names, aliased subqueries, CTEs, UNION queries.
- **Solution:** Use sqlparse AST to properly identify all table references.

### #18 No Pagination for Large Results
- **Files:** `src/queryease/formatter.py`, `main.py`, `src/queryease/generator.py`
- **Problem:** Results are truncated at 50 rows with a warning, but there's no way to see the rest.
- **Solution:** Add `--page N` flag; implement interactive pager; add LIMIT/OFFSET suggestion.

### #19 No Export Options for Results
- **Files:** `src/queryease/formatter.py`, `main.py`
- **Problem:** Results only display in terminal. No CSV, JSON, or file export option.
- **Solution:** Add `--format csv|json|table` option; add `--output filename` flag.

### #20 Unclear Error Messages for LLM Failures
- **Files:** `src/queryease/generator.py`, `src/queryease/formatter.py`
- **Problem:** When Groq API fails, users see generic "SQL generation failed" messages without actionable guidance.
- **Solution:** Provide specific error messages for rate limits, auth failures, timeouts.

### #21 No Schema Discovery Command
- **Files:** `main.py`, `src/queryease/schema.py`
- **Problem:** Users can't easily list tables or explore schema without running a query.
- **Solution:** Add `--schema` or `--tables` flag to list available tables and columns.

### #22 No Way to See Generated SQL Without Execution
- **Files:** `main.py`
- **Problem:** SQL is always executed after generation. No "dry run" mode.
- **Solution:** Add `--dry-run` or `--sql-only` flag to show SQL without execution.

### #26 Schema Cache Validation Inefficiency
- **Files:** `src/queryease/schema.py`
- **Problem:** Cache validation fetches the entire live schema from DB to compute hash on every cache check.
- **Solution:** Cache the schema hash separately; use incremental schema change detection.

### #27 No Connection Pooling
- **Files:** `src/queryease/db/*.py`, `src/queryease/executor.py`
- **Problem:** Each query opens and closes a new database connection.
- **Solution:** Implement connection pooling; reuse connections across queries within a session.

### #28 No Query Timeout Configuration
- **Files:** `src/queryease/executor.py`, `src/queryease/db/*.py`, `src/queryease/config.py`

### #30 No Caching of LLM Responses

- **Solution:** Cache LLM responses by question hash; implement semantic caching.

### #31 Tight Coupling to Groq API
